02Розвилка — PWA vs нативний агент

Найцінніша персона (B) хоче контекстний автотрекінг (VS Code / Git / Jira / Slack), а він вимагає нативного агента — браузер бачить ~20–30% дня. Наш продукт — PWA. Це не баг ресерчу, а межа платформи (research §2 / G2).

1 · Чистий PWA (поточний курс)

Обслуговуємо A і B частково: найшвидший ручний таймер + 1-tap confirm + idle-trim + invoice-draft. Свідомо не граємо в автотрекінг. Дешево, чесно з local-first; стеля по B обмежена.

2 · PWA + нативний компаньйон

Desktop-агент лише на метаданих (активний застосунок, назва репо/workspace, URL тикета, час активності). Закриває ядро B, але це велика робота + хмара/синк + суперечить «без-бекенду MVP».

3 · Гібрид у часі

MVP = шлях 1; нативний агент — окремий пост-MVP трек, якщо валідація B масштабується (поки N=1).

📌 Поки не міняємо CLAUDE.md (MVP = шлях 1). Перед рішенням: ще 2–3 інтерв'ю фултайм-девів + оцінка обсягу агента. Якщо шлях 2/3 — діє межа метадані можна, контент ніколи, без вивантаження контенту в хмару.

03Чесні слабкі місця

🔴 Найнебезпечніший список — керують продуктом, стоять на [?] / вигаданому

1. Персона A як PRIMARY. Уся пріоритизація зважена на side-gig дизайнера — бо це перші дані, не бо доведено, що це найцінніший ринок. Стоїть на 2 людях.

2. «Простота — незакритий гэп». Власні дані підточують: I#2 та I#3 задоволені Toggl і звуть його простим. Категорія місцями вже «good enough».

3. «Desktop-first → геть mobile» — вже змінило CLAUDE.md на N=3 з перекосом у дизайнерів.

4. Персона C + «не робити автотрекінг». Тримається на архітектурі (міцно) + вигаданій C + 1 особі (I#2).

5. R4 у ядрі при важливості A=2. Інвойсинг у ядрі, хоча для primary це «дратує», не must-have; тяга гэпу до вибору не доведена.

6. Idle-trim як рішення R2 — у research позначено [?] (Page Visibility API у фоні PWA). Якщо не злетить — функція порожня.
