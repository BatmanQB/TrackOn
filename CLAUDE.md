# TrackOn — Product & Engineering Brief

> Документ-джерело істини для продукту. Оновлюй його, коли змінюються рішення.
> Мова продукту: українська (UI), код/коментарі — англійською.

## 1. Суть продукту

**TrackOn** — кросплатформний веб-застосунок (адаптивний PWA) для трекінгу часу по проєктах
для фрілансерів. Мета: **максимально швидко затрекати час**, бачити загальні витрати
часу й грошей, формувати звіти за період та інвойси з мінімальним інтерфейсом.

### Цільова аудиторія
Фрілансери, які:
- трекають час по кількох клієнтах/проєктах,
- хочуть бачити звіти й статуси оплати,
- формують інвойси,
- цінують простий і швидкий інтерфейс понад усе.

### Головний принцип
**Швидкість і простота.** Затрекати час — у 1-2 тапи з головного екрана.
Кожна фіча перевіряється питанням: «чи не ускладнює це базовий сценарій трекінгу?»

### Хто і навіщо (з ресерчу — деталі у [`research/personas.md`](research/personas.md), [`research/jtbd.md`](research/jtbd.md))

**Primary-персона (A):** фрілансер-професіонал (UX/UI-дизайн / front-end), для якого фріланс
часто **додатковий** дохід; веде 1–3 клієнти, трекає **з комп'ютера**, цінує простоту понад усе
й болісно реагує на складність та paywall. *(B — фултайм-розробник із високою готовністю платити —
це **платник**, монетизацію цілимо в нього; деталі — personas.md §7.)*

**Main job:** «Коли я виконую оплачувану роботу для кількох клієнтів, я хочу без зусиль
зафіксувати, куди дівся мій час, щоб упевнено отримати справедливу й безсуперечну оплату.»

**Топ-3 jobs у ядро MVP** (важливі для primary + ринок не закриває):
1. **R1 — почати облік, не втрачаючи фокус** (старт у 1–2 дії).
2. **E2 — не загубитися у складності** (мінімалізм як гігієна; обережно — Toggl уже простий).
3. **R4 — час → оплата без ручного зведення** (білінг без paywall у простому локальному інструменті).

## 2. Платформа

- **Адаптив, desktop і mobile рівноцінно.** Первинний контекст трекінгу — **комп'ютер**
  (підтверджено Interview #1, [`research/personas.md §6`](research/personas.md)); телефон —
  додатковий сценарій. Тому **mobile-first як позицію знято** (раніше було гіпотезою H2).
- **Встановлюваний PWA**: працює як застосунок на десктопі й телефоні, **офлайн-first**.
- Великі зручні цілі для тапа/кліку, мінімум кроків для частих дій.

## 3. Архітектура даних

- **Етап 1 (зараз): local-first.** Усі дані локально в **IndexedDB** (через Dexie.js).
  Працює офлайн, без бекенду, без акаунтів. Один локальний користувач.
- **Етап 2 (пізніше): хмара + синхронізація.** Підключення **Supabase**
  (Postgres + Auth + Realtime) для синку між пристроями. **Вхід опційний** — застосунок
  стартує локально без акаунта, логін лише для синку (без реєстр-стіни на старті).
  Мультидевайс — підтверджена потреба ([`research/personas.md §6`](research/personas.md)).
  Відомий компроміс free-тарифу Supabase: проєкт засинає після ~тижня неактивності.
- **Вимога до коду:** шар доступу до даних (repository) ізольований від UI, щоб додати
  синк без переписування інтерфейсу. Усі сутності мають `id` (UUID), `createdAt`,
  `updatedAt`, м'яке видалення (`deletedAt`) — щоб майбутній синк був безконфліктним.

## 4. Модель даних

```
Client (Клієнт)
  └── Project (Проєкт)
        └── TimeEntry (Запис часу)
```

### Client
- `id`, `name`, `archived`, `createdAt`, `updatedAt`, `deletedAt`
- Реквізити для інвойсу (назва, контакт) — опційно.

### Project
- `id`, `clientId`, `name`, `color`
- `currency`: `UAH | USD | EUR` (дефолт `UAH`)
- `hourlyRate`: number **nullable** — якщо `null`, проєкт трекає лише час (без грошей)
- `archived`, `createdAt`, `updatedAt`, `deletedAt`

### TimeEntry
- `id`, `projectId`
- `startedAt`, `endedAt` (nullable, поки таймер активний)
- `durationSec` (обчислюється; ручні записи задають напряму)
- `note`: коментар до запису (опційно)
- `rateOverride`: number nullable — перевизначає `project.hourlyRate` для цього запису
- `source`: `timer | manual`
- `createdAt`, `updatedAt`, `deletedAt`

### Invoice
- `id`, `clientId`, `currency`, `periodFrom`, `periodTo`
- `entryIds`: список включених записів
- `total` (час × ставка)
- `status`: `unpaid | sent | paid` (не сплачено / відправлено / сплачено)
- `createdAt`, `updatedAt`

## 5. Трекінг часу

- **Таймер + ручне введення** (обидва способи).
- **Таймер:** Start/Stop у реальному часі. **Один активний таймер** одночасно.
  Таймер **переживає перезавантаження сторінки й офлайн** (стан персиститься).
- **Ручне введення:** додати/відредагувати запис (час від-до або тривалість),
  на випадок «забув увімкнути таймер».
- До будь-якого запису можна додати **коментар**.

## 6. Гроші та ставки

- Ставка задається **на рівні проєкту** (`hourlyRate`), **необов'язкова**.
- **Оверрайд ставки** можливий на рівні окремого запису (`rateOverride`).
- Якщо ставки нема → запис трекає **лише час**, без суми.
- **Валюта — на рівні проєкту** (`UAH | USD | EUR`).
- **Агрегація на дашборді групується по валюті** — без конвертації курсів у MVP.
  (Не сумуємо різні валюти в одне число.)

## 7. Звіти

- **Тільки на екрані** (без CSV/PDF-експорту звітів у MVP).
- Періоди: **тиждень / місяць / довільний діапазон**.
- Розрізи: по проєкту та по клієнту.
- Показники: сумарний час + сума грошей (по кожній валюті окремо).

## 8. Інвойси

- Генерація **PDF** з обраних записів за період.
- **Одна валюта на інвойс** (валюта проєкту/клієнта).
- **Статуси:** не сплачено → відправлено → сплачено.

## 9. Дизайн

- **Мінімалізм.** Чистий інтерфейс, великі тап-зони для телефону.
- **Теми light + dark**, авто за системними налаштуваннями + ручний перемикач.
- Нейтральна палітра з однією акцентною кольоровою — **пілотовано як мох-зелений
  `#4a710a`** у напрямі «Мох» (див. «Концепт» нижче); фінальне затвердження — при
  перенесенні в `tokens/`.

### Концепт

**Джерело істини — [`concept/taste.md`](concept/taste.md)** (Смак дизайнера · Атрибути ·
Напрями · Аудит). Обраний візуальний напрям — **«Мох»**: тихий, паперово-білий, один
моховий акцент (не «зелений = гроші»), геометричний sans-serif Outfit + Inter, радіус
картки 20px, іконки Solar Linear, light/dark як одна токен-система (не інверсія).

- Кожне візуальне рішення підписане атрибутом (з якого рядка даних/смаку воно складене) —
  правило «нічого з голови» перевіряється прискіпливими аудитами (запис у `taste.md §Аудит`).
- **Пілотний стенд** (жива палітра, типографіка, форма, план іконок, компоненти) —
  [`concept/concept.html`](concept/concept.html); **два записані альтернативні напрями**
  (Термінал, Людяний) — [`concept/directions.html`](concept/directions.html); **джерела
  прийомів** (Acctual, Runey, monobank, Popcorn…) — [`concept/references.md`](concept/references.md).
- Мова перенесена 1:1 у вайрфрейми `wireframes/` (той самий токен-набір, ті самі радіус/
  іконки/статус-бейджі у списку й деталі) — жоден вайрфрейм не вигадує власне рішення.

### Голос (Voice)

**Джерело істини — [`voice.md`](voice.md)** (Принципи · Словник · Заборонене · Мікрокопі).
Увесь текст UI пишемо за ним; словник термінів і перевірку розбіжностей ведемо в [`microcopy.md`](microcopy.md).

- **5 принципів:** (1) говоримо до **однієї людини** («ти»), не до «команди»; (2) гроші —
  **конкретна сума** ($120 · ₴1 200), не «прибутковість»; (3) **без хайпу** — кажемо, що сталося;
  (4) **чесно про ручний ввід**, не «AI зробить сам»; (5) **людські слова**, не жаргон обліку.
- **Звертання:** тільки **«ти»**, з малої, однаково на всіх екранах.
- **Словник (одне поняття — одне слово):** «інвойс» (не «рахунок»), «+ Проєкт»/«+ Клієнт»
  (не «+ Новий…»), «Зберігаємо…» (не «Збереження…»), «Тиждень · Місяць · Діапазон»,
  «оплачувані години» (не «білабельний»), «своя ставка» (не «оверрайд»), «Таймер іде /
  не запущено», «Генеруємо PDF…», «Назад до {місце}», мітки «Відстежено:» / «Зароблено:».
- **Дозволені англіцизми:** таймер, Старт, Стоп, PDF, фрілансер, інвойс. **Заборонені:**
  білабельний, оверрайд, автодобір.
- **Заборонено:** AI-кліше («Ой, щось пішло не так»), вітання, мотиваційний тон, зайві оклики,
  емодзі в системних повідомленнях, слово «успішно», TODO-примітки.
- **Тон станів:** порожній — веде до дії; помилка — що сталося + що робити (без вибачень/жартів);
  завантаження — мовчить або «дієслово в процесі»; успіх — факт + крок, без святкування.

## 10. Технологічний стек

- **Frontend:** React + TypeScript + Vite
- **Стилі:** Tailwind CSS
- **PWA:** vite-plugin-pwa (service worker, офлайн, install prompt)
- **Локальна БД:** IndexedDB через **Dexie.js**
- **Стейт:** легкий store (Zustand) + реактивні запити Dexie
- **Графіки звітів:** легка бібліотека (напр. Recharts) — за потреби
- **PDF інвойсів:** клієнтська генерація (напр. pdf-lib / jsPDF)
- **Майбутній бекенд:** Supabase (Postgres + Auth + Realtime) на Етапі 2

## 11. Scope MVP (Етап 1)

**В обсязі:**
- CRUD клієнтів і проєктів (зі ставкою, валютою, кольором)
- Таймер (Start/Stop, персистентний) + ручні записи з коментарями
- Дашборд: загальний час і гроші (по валютах)
- Звіти на екрані (тиждень/місяць/період, по проєкту/клієнту)
- Інвойси: вибір записів → PDF + статуси оплати
- PWA: інсталяція, офлайн
- Light/dark теми

**Поза обсягом MVP (пізніше):**
- Акаунти, авторизація, хмарна синхронізація (Етап 2)
- Конвертація валют за курсом
- CSV/PDF-експорт звітів
- Командна робота / кілька користувачів
- Інтеграції (календар, платіжки)

## 12. Відкриті питання

- Акцентний колір **пілотовано** (мох-зелений `#4a710a`, напрям «Мох» — §9 «Концепт»);
  залишається формально затвердити при перенесенні в `tokens/`. Логотип — відкрито.
- Реквізити для шапки інвойсу (дані виконавця): задавати в налаштуваннях.
- Чи потрібні теги/категорії записів (наразі поза MVP).
